# AGENTS.md

Guidance for OpenCode sessions working in this repo. Read `docs/design.md` before
changing core flow — many "obvious improvements" are explicit non-goals (see §8).

## Agent usage policy

**Use available agents for all substantive work.** This repo has specialized agents
(`QATestEngineer`, `explore`, `UIUXDesigner`, etc.). The orchestrator agent should:

- **Always delegate** to the appropriate agent rather than writing code directly.
  Only write code yourself when no suitable agent exists or the change is trivial
  (1–3 lines with zero logic).
- **Partition work by file** (see below) when dispatching multiple agents.
- **Verify** that the agent completed the work correctly (run builds/tests).
- **Read `AGENTS.md` before starting** — this file is the single source of truth
  for build commands, conventions, and boundaries.

## Multi-agent coordination

When dispatching multiple agents in parallel, partition work by **file ownership** to
prevent edit conflicts:

- **Agents work only inside the project root directory.** Don't read or write outside
  the repo workspace.
- **One file, one owner.** Each file is owned by exactly one agent for the duration of
  the task. No two agents may edit the same file concurrently. Assign every target file
  to a single agent in the task prompt.
- **Partition by file, not by region.** Don't ask two agents to edit different regions
  of the same file (e.g., "agent A owns the top bar, agent B owns the bottom bar" of one
  screen). Region-level splits cause merge clobbering and dropped imports — partition at
  the file level instead. If two features touch one file, one agent does both.
- **Shared helpers go to one owner.** If both agents need a new shared composable/util,
  assign its creation to exactly one agent; the other consumes it read-only or waits for
  it to land.
- **State the ownership map in each task prompt.** Explicitly list which files each
  agent may edit and which are off-limits.

## Build & environment

- Single-module Android app (`:app`), Kotlin + Jetpack Compose. JDK 17, AGP 8.1.4,
  Kotlin 1.9.22, Compose compiler 1.5.8, compileSdk/targetSdk 34, minSdk 26.
- Dev environment is managed by [devenv](https://devenv.sh) (Nix). **`local.properties`
  is auto-generated by nix-shell** and gitignored — never commit or hand-edit it; its
  `sdk.dir` points into the Nix store and is machine-specific.
- Enter the dev shell with `devenv shell`. Exposes scripts: `build`
  (`./gradlew assembleDebug`), `install` (`./gradlew installDebug`), `clear`
  (wipes `local.properties` — use if SDK path is stale).
- SDK / build-tools come from `devenv.nix`. Don't install via Android Studio's SDK
  Manager without updating `devenv.nix`, or paths will drift.
- No CI, no pre-commit hooks. Verification is manual.

## Commands

- Build: `./gradlew assembleDebug`
- Run JVM unit tests: `./gradlew testDebugUnitTest` (release variant: `testReleaseUnitTest`)
- Single test class: `./gradlew testDebugUnitTest --tests "org.notifledger.app.parser.ParserEngineTest"`
- Single test method: `./gradlew testDebugUnitTest --tests "*.ParserEngineTest.parse*"`
- Install on device: `./gradlew installDebug`
- No `lint`/`detekt`/`ktlint` is configured. AGP's built-in `./gradlew lintDebug` runs
  but isn't part of the workflow; there is no CI to gate on it.

## Tests

- Tests are **JVM unit tests only** (`app/src/test/...`, plain JUnit 4). No Robolectric,
  no MockK/Mockito, and `app/src/androidTest` is empty — there are no instrumented tests.
- Anything needing Android framework classes must be unit-testable without a device, or
  restructured. Don't assert on `Context`/`Uri`/framework types in unit tests without
  adding Robolectric first.
- Test packages mirror `main`: `parser`, `journal`, `notification`, `settings`.

## Architecture (deliberately non-standard)

NotifLedger turns bank/payment notifications into [hledger](https://hledger.org)
journal entries automatically. Core stance: if a bank told you it happened, don't make
the human confirm it.

- **No DI framework.** Manual construction in `NotifLedgerApp` (the `Application`).
  Don't introduce Hilt/Koin — it's an explicit non-goal (design.md §8).
- **No Room / database.** State lives in the hledger journal file (via Storage Access
  Framework `content://` URIs) and app settings in DataStore Preferences. Don't add a
  DB "to cache things" — that's a non-goal.
- **`JournalWriter` is pure** — no file I/O. The ViewModel reads/writes through Android's
  `ContentResolver`. Keep I/O out of `JournalWriter`; it's tested as a pure text
  formatter and that boundary matters.
- **`NotifLedgerApp` uses `runBlocking`** to read settings synchronously for the
  `NotifListener` path — intentional, not a bug to "fix" with coroutines.
- **Parser is intentionally dumb**: first number in the notification = amount, title =
  payee. Norwegian comma decimals (`184,50`) handled. Do **not** add per-bank parser
  config — explicit non-goal (design.md §8).
- KSP plugin is declared in root `build.gradle.kts` but **not applied in `:app`** (no
  Room/Hilt/Moshi). Don't assume KSP annotations work without applying the plugin.

## Notification flow

- `NotifListener` extends `NotificationListenerService` (manifest-declared, protected
  permission) — the core entry point, wired in `AndroidManifest.xml`.
- Capture is **allowlist-filtered by package name** (user-selected apps), not global.
  Allowlist lives in DataStore via `SettingsManager`.
- `BIND_NOTIFICATION_LISTENER_SERVICE` is a protected permission — the user must grant
  it in system Settings (Notifications → Notification access). Nothing in-app can grant
  it silently; don't try.
- `NotificationHelper.showListeningNotification` posts a persistent foreground-style
  notification on startup — required to keep the listener alive, don't remove it.

## Style conventions (repo-specific)

Source of truth: `docs/coding_standarts.md`. Points that diverge from defaults:

- **Simple beats clever.** Write the dumbest version that works; no abstraction
  layers/factories until 3+ concrete cases need it. Don't build for hypothetical futures.
- **One function, one action.** If you describe it with "and", split it. Max ~3 levels
  of nesting; extract the inner logic.
- **Comments explain *why*, never *what*.** If the *what* isn't obvious, rewrite the
  code — don't caption it. Delete commented-out code.
- **Feature-based package structure**, not type-based: `parser/`, `journal/`,
  `notification/`, `ui/`, `model/` — group by what code *does*, not by
  `controllers/`/`models/`/`views/`.
- **Fail loudly and early.** No silent catches that swallow errors. (Note:
  `NotifLedgerApp` does catch-and-default in a few `runBlocking` blocks — those are
  deliberate boundary fallbacks, not a pattern to spread.)
- No file should require scrolling to understand its purpose. Optimize for the next
  reader, including future you.

## Before opening a PR

From `docs/coding_standarts.md`:

1. Could someone unfamiliar with this code understand it in one read?
2. Does every function do exactly one thing?
3. Is there anything "just in case" that isn't used yet? Delete it.
4. Could this be simpler and still work? If yes, it's not done yet.
